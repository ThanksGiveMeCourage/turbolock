## 二、四个攻坚目标

### 2.1 池化 rand buffer（-2 allocs）

```go
// 当前（每次 Lock 分配 32 字节）
b := make([]byte, 32)
rand.Read(b)

// 目标（sync.Pool 复用）
var randPool = sync.Pool{
    New: func() any { return make([]byte, 32) },
}

b := randPool.Get().([]byte)
rand.Read(b)
defer randPool.Put(b)
```

**注意事项**：
- `base64.StdEncoding.EncodeToString(b)` 会创建新的 string，当前 `b` 放入 Pool 后不能再引用其内容
- 需要先 `EncodeToString` 得到最终 value，再 `Put(b)`

### 2.2 池化 timerTask 节点（-1 alloc）

```go
// 当前（每次 addTask 分配一个 timerTask）
task := &timerTask{...}

// 目标（sync.Pool 复用）
var taskPool = sync.Pool{
    New: func() any { return &timerTask{} },
}

task := taskPool.Get().(*timerTask)
task.key = key
task.callback = callback
task.ticks = int(interval / tickMs)
task.cancelled = false
// addTask 完成后不放回——任务生命周期由时间轮管理
// 在 fireSlot 惰性清理时放回
```

**关键点**：
- `fireSlot` 中 `cancelled` 任务的清理处，加上 `taskPool.Put(task)`
- `removeTask` 被调用后，任务最终在 `fireSlot` 被清理时才放回
- 注意 `callback` 闭包引用可能阻止 GC，清理时需设为 nil

### 2.3 逃逸分析排雷

```bash
go build -gcflags="-m" ./... 2>&1 | grep "escapes to heap"
```

需要让以下变量**尽可能锁死在栈上**：

| 变量 | 当前状态 | 目标 |
|------|:--:|:--:|
| `value` (base64 string) | 堆（传给闭包） | 栈（改为参数传值） |
| `key` (string) | 堆（传给闭包） | 同上 |
| `timerTask` | 堆（new） | 池化后变池 |
| 续期闭包 | 堆（函数字面量） | 方法值替代 |

### 2.4 go-redis 内部分配（不可消除，但可压低）

go-redis 的 `SetNX` 和 `Eval` 内部会分配 Cmd 对象、序列化缓冲等，这些是第三方库的硬开销。只能通过连接池预热 (`Ping` + `PoolSize`) 压低，无法消除。

## 七、核心认知

### 7.1 零分配 = TurboLock 自身零逃逸

go-redis 作为网络 I/O 库，必然会创建序列化缓冲区和 Cmd 对象。**阶段四的目标是消除 TurboLock 自身代码的堆逃逸**，第三方库的分配属于"硬成本"。

经逃逸分析验证：`make([]byte, 32)` 和 `&timerTask{}` 均已从逃逸列表中消失。allocs/op 未显著下降是因为 go-redis 占 ~6/10 的分配量，TurboLock 自身节省的 ~2 allocs 被其波动掩盖。

### 7.2 Pool 的正确关闭

```
错误: 用完不放回 → 内存泄漏（池子永远在申请新的）

正确: 
  Get → 使用 → Put（精确配对）
  fireSlot 中 cancelled 清理时 Put
  timerTask 的 callback 置 nil 后再 Put（防止闭包引用阻止 GC）
```

### 7.3 不要过早优化

如果某项池化（如 `base64.Encoding`）带来的收益 < 1 alloc/op，且代码复杂度增加 > 5 行，则放弃。**阶段四的目标是 11 → 0~3，不是绝对的 0。**
