# 为什么分布式锁续期间隔是 2.6 秒？

> 相关源码：`turbolock_impl.go`（续期逻辑）、`timing_wheel.go`（时间轮调度）  
> 一句话答案：**2.6s = 8s（默认 TTL）÷ 3，是 Redis 社区公认的续期黄金比例。**

---

## 一、2.6 秒从哪来？

它不是拍脑袋选的，而是从三个确定性条件推导出来的：

```
条件 ①：锁的默认过期时间 = 8 秒（TurboLock Options.Expiry 默认值）
条件 ②：续期必须在 TTL 到期前完成
条件 ③：续期不能太频繁（浪费 Redis 资源），也不能太稀疏（容易错过）
条件 ④：业界最佳实践 = TTL / 3（Redisson、RedSync 等主流库通用）

→ 8s / 3 ≈ 2.666 秒 → "2.6 秒"
```

代码中就是一行：

```go
renewInterval := t.opts.Expiry / 3   // 8s / 3 ≈ 2.666s
```

---

## 二、为什么是 TTL/3，而不是 TTL/2 或 TTL/4？

### 可视化对比

```
TTL = 8s 的时间轴

TTL/2 = 4s（续得太晚）
  ┌──────────────┬──────────────┐
  │   续期       │   ！！危险   │
  0s             4s             8s
                  ↑                
        距过期只差 4s，一次网络抖动就凉了

TTL/3 = 2.6s（黄金比例）
  ┌──────┬──────┬──────┬──────┐
  │ 续期 │ 续期 │ 续期 │ 安全 │
  0s    2.6s   5.3s   8.0s   ...
          ↑              
  每次续期把 TTL 重置为完整的 8s
  有 2.6s 的安全缓冲，绰绰有余

TTL/5 = 1.6s（续得太勤）
  ┌──┬──┬──┬──┬──┬──┬──┬──┐
  续期续期续期续期续期续期...
  0s  1.6s 3.2s 4.8s 6.4s 8.0s
  Redis 压力翻倍，但安全收益没有增加
```

### 定量分析

| 续期频率 | 续期次数/8s | 安全边界 | Redis 压力 | 综合评价 |
|---------|:--:|:--:|:--:|:--:|
| TTL/2 (4s) | 2 | **2s**（危险） | 低 | ❌ 不安全 |
| **TTL/3 (2.6s)** | **3** | **2.6s**（安全） | 适中 | ✅ **最佳** |
| TTL/4 (2s) | 4 | 4s | 偏高 | 🟡 过于保守 |
| TTL/5 (1.6s) | 5 | 6.4s | 高 | ❌ 浪费 |

## 三、2.6 秒进入时间轮后变成了什么？

时间轮的 tick 间隔是 100ms，所以 2.6s 要转换成"格子数"：

```
2600ms ÷ 100ms = 26 个 tick
```

然后**整个过程就是 `26` 这个数字在流转**：

```
addTask(2.6s, callback)
  │
  ▼
ticks = 26                          ← 翻译成 26 个格子
  │
  ▼
insertTask:                         ← 决定贴在哪个槽位
  26 < 256 → 走 Level 0
  slot = (cursor0 + 26) % 256       ← 贴在"26 格之后"的位置
  │
  ▼
时光流逝... 每 100ms 指针走一格
  │
  ▼
2.6 秒后，指针到达目标槽位
  │
  ▼
fireSlot:                           ← 取出纸条，执行任务
  unlink → go callback()            ← 异步调用 Redis EXPIRE
  insertTask(task)                  ← 重新贴到"26 格之后"
  │
  ▼
循环往复，每隔 2.6 秒续一次
```

---

## 四、为什么 26 < 256 很重要？

时间轮有三层，每层的容量不同：

```
Level 0: 256 格 → 能容纳 0 ~ 25.6 秒的任务
Level 1:  64 格 → 能容纳 25.6 秒 ~ 27.3 分钟的任务
Level 2:  64 格 → 能容纳 27.3 分钟 ~ 29.1 小时的任务
```

**26 远小于 256，所以续期任务永远停在 Level 0**：

- 不需要级联降级（Level 1 → Level 0）
- 不需要更复杂的数据结构操作
- O(1) 的数组索引 + 链表插入，纳秒级完成

这是有意设计的——**主力场景（2.6s）用最快的路径，Level 1/2 只是兜底保险**。

## 五、如果用户改了 TTL 呢？

```go
// 用户自定义
WithExpiry(30 * time.Second)

// Lock() 中运行时计算
renewInterval := t.opts.Expiry / 3   // 30s / 3 = 10s
ticks := int(10s / 100ms)           // = 100
```

```
TTL    RenewInterval    ticks    在哪层？
 8s  →     2.6s     →    26   →  Level 0
30s  →      10s     →   100   →  Level 0
60s  →      20s     →   200   →  Level 0
90s  →      30s     →   300   →  Level 1（首次触发级联！）
```

只要 TTL ≤ 76s，续期间隔 ≤ 25.6s，就永远在 Level 0。绝大多数业务场景的锁 TTL 都在 60s 以内，所以级联几乎不会触发。

## 六、与主流库的对比

| 库 | 语言 | 默认 TTL | 续期间隔 | 比例 |
|----|------|:--:|:--:|:--:|
| **TurboLock** | Go | 8s | 2.6s | **TTL/3** |
| Redisson | Java | 30s | 10s | **TTL/3** |
| RedSync | Go | 8s | 手动续期 | 无内置 |
| redis-lock (Node) | JS | 10s | TTL/2 | TTL/2 |
| etcd | Go | 60s | TTL/3 (lease) | **TTL/3** |

**TTL/3 不是巧合，是分布式系统领域经过大量实践验证的经验值。**

---

## 七、核心要点

| 关键点 | 值 |
|--------|-----|
| 续期间隔 | `Expiry / 3` |
| 默认 TTL | 8s |
| 默认续期间隔 | **2.6s** |
| tick 翻译 | **26 ticks** |
| 所在层级 | **Level 0**（永远不用级联） |
| 来源 | Redisson / RedSync / etcd 的通用实践 |
| 为什么不用 TTL/2 | 安全边界太薄，网络抖动易错过 |
| 为什么不用 TTL/4 | Redis 压力翻倍，安全收益不增加 |
