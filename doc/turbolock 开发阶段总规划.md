# turbolock 开发阶段


## 阶段一：建立骨架与基准（The Baseline）

核心目标： “先上车，后补票”。用最糙、最稳的代码把项目架子搭起来，跑通黑盒，并在本地钉上“性能耻辱柱”。

* 1.1 接口与配置定义： 在完全不写实现的情况下，把对外暴露的 TurboLocker 接口、UnlockFunc 闭包函数以及高大上的 Functional Options 配置模式（仿照 redsync）手敲出来。
* 1.2 单点核心功能组装： 砍掉 redsync 的多节点红锁死循环，提取其精华（SETNX 抢锁、Lua 脚本防误删释放、crypto/rand 生成随机值），手写一个只连接单台 Redis 的最简版分布式锁。
* 1.3 搭建压测靶场（Benchmark）： 引入 go test -bench 脚本。开启 10,000 个 Goroutine 在本地疯狂抢锁，记录下此时糟糕的 CPU、网络 I/O 以及内存分配次数（allocs/op）。

## 阶段二：攻坚大厂面试核心 —— 单机合流技术（The Coalescing）

核心目标： 解决网关雪崩与缓存击穿痛点。不碰 Redis，纯粹在 Go 语言的运行时（Runtime）层面玩高并发控制。

* 2.1 局部锁槽设计： 在 TurboLocker 结构体内部引入基于 Key 哈希的分段本地锁（如 sync.Map 或自定义 Mutex 数组）。
* 2.2 编织挂起与广播唤醒： 当 10,000 个相同 Key 的请求涌入时，利用 Go 底层的 sync.Cond（条件变量）或者 channel 拦截它们。只放 1 个协程去调 Redis，其余 9999 个协程在本地原地休眠，等待广播唤醒。
* 2.3 降维打击跑分： 再次运行 Benchmark。你会亲眼看到远程 Redis 的 QPS 暴跌 90% 以上，本地高并发吞吐量飙升。

## 阶段三：攻坚性能黑洞 —— 全局层级时间轮看门狗（The Timing Wheel）

核心目标： 解决分布式锁的自动续期（Extend/Touch）问题，彻底打破“一锁一协程”的性能瓶颈。

* 3.1 摒弃传统做法： 坚决不用 go func() { time.Sleep() } 这种低级续期方案，避免高并发下协程与内核定时器（Timer）爆炸。
* 3.2 手撕单机时间轮： 利用双向链表和 CAS 无锁/轻量锁操作，手写一个层级时间轮算法。整个系统只用一个全局单协程的 Timer 作为心跳，驱动指针旋转。
* 3.3 异步续期挂载： 抢锁成功后，将锁的上下文作为节点，动态投喂进时间轮；锁释放时，从时间轮中安全剥离。

## 阶段四：压榨性能至极限 —— 零内存分配（Zero-Allocation）

核心目标： 真正向顶级开源库靠拢。利用 Go 的编译期特性和黑盒工具，将高并发核心路径的内存分配数压榨到 0。

* 4.1 逃逸分析排雷： 开启 go build -gcflags="-m"，像雷达一样扫描整个核心链路。找出所有因为指针乱用、闭包参数逃逸到堆（Heap）上的变量，强行把它们锁死在栈（Stack）上。
* 4.2 引入 sync.Pool 对象池： 对于无法避免要频繁创建的 Context 包装结构体、临时通信 Meta 块，接入 sync.Pool 进行高速复用。
* 4.3 终极跑分： 压测报告上的 allocs/op 变为完美的 0。

## 阶段五：开源发布与知识变现（The Show）

核心目标： 把你的技术成果转化为不可磨灭的“个人技术资产”和“大厂面试通行证”。

* 5.1 编写“攻击性”README： 整理出阶段一、二、四的 Benchmark 性能对比图表，用数据说话。
* 5.2 编写大厂级“故障复盘/架构演进”技术博客： 详细记录你如何发现冗余、如何通过单机合流与时间轮打破性能瓶颈的心路历程。
* 5.3 社区输出： 丢到 GitHub、即刻、掘金等技术圈，迎接真实的反馈。
