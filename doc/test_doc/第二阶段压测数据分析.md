* 为什么单机合流不仅没快，反而引发了“大惨案”？
* 结合你的代码逻辑，病因其实非常明确，主要有两个致命点：
* * 1、致命缺陷 1：全局大锁（globalMu）的锁竞争
* * * 现象： 10000 个并发请求，在调用 getSlot 时，全都需要去抢夺唯一的一把 globalMu。
* * * 本质： 这是一种极其严重的“锁争用（Lock Contention）”。把原本可以并行的 10000 个请求，强行压扁成了一个单行道，导致 CPU 疯狂进行上下文切换，单次耗时飙升。
* * 2、致命缺陷 2：“护航效应”导致的串行送死（葫芦娃救爷爷）
```go
for slot.active {
    slot.cond.Wait()
}
slot.active = true // Leader 归来唤醒大家后，下一个醒来的协程又把门关上了！

defer func() {
    slot.mu.Lock()
    slot.active = false
    slot.cond.Broadcast() // 广播，唤醒等待队列中的所有 goroutine
    slot.mu.Unlock()
}()

value, err := t.getValue()
if err != nil {
    return nil, err
}
// 2. 经典的重试退避循环
for i := 0; i < t.opts.Tries; i++ {
    //....
    // 抢锁成功，构建并返回释放锁的闭包
    return func(unCtx context.Context) error {
        // 执行 Lua 脚本原子释放
        return t.client.Eval(unCtx, delLuaScript, []string{key}, value).Err()
    }, nil
}

return nil, ErrLockFailed
```
* * 灾难时序：
* * * 1、Leader 抢到锁，唤醒所有人，自己退出了。
* * * 2、第一个被唤醒的 Follower 发现门开了，它又把门关上（active = true），然后兴冲冲跑去 Redis 抢锁。
* * * 3、此时 Redis 的排他锁还在前任 Leader 手里，所以这个 Follower 必然抢不到，它被迫进入 100 次的重试循环，死死卡住几百毫秒。
* * * 4、等它终于绝望放弃后，再唤醒下一个 Follower，下一个 Follower 继续把门关上，再去 Redis 撞墙……
* * 本质： 这根本不是合流，而是把原本可以在 1 毫秒内并发失败的 9999 个请求，变成了排着队、一个个去 Redis 面前“排队枪毙”，导致耗时直接拉长了 10000 倍。

* 其中，【此时 Redis 的排他锁还在前任 Leader 手里，所以这个 Follower 必然抢不到，它被迫进入 100 次的重试循环，死死卡住几百毫秒。】需要解读一下
* * 这里存在的问题是：**潜意识里把“G1（Leader）唤醒大家”和“G1 去 Redis 里把锁删掉”当成了同一时间发生的事情。**
* * 但实际上，在代码的物理执行层面，这两件事之间横亘着一道巨大的“时间鸿沟”。
* * 四个角色的时间线交错：
* * * 假设我们有两个协程：G1（前任 Leader） 和 G2（被唤醒的 Follower）。
* * * 配置项为：重试 100 次，每次间隔 10ms（time.After(10ms)）。
* * 第一幕：G1 凯旋，打开城门
* * * [T=0ms] G1 向 Redis 发送 SetNX，成功抢到排他锁。
* * * [T=1ms] G1 准备 return UnlockFunc, nil，在 return 之前，它触发了 defer。
* * * [T=2ms] G1 在 defer 里执行 slot.active = false（打开本地城门），并执行 Broadcast() 唤醒了睡梦中的 G2。
* * * [T=3ms] G1 的 Lock 函数正式执行完毕，把 UnlockFunc 交给了外层的业务代码。
* * 注意：
* * * 此时此刻（T=3ms），G1 只是在本地退出了 Lock 函数，它并没有释放 Redis 里的锁！ > 业务代码才刚刚拿到锁，准备开始干活（比如去查数据库、写文件），可能需要耗时 200ms 甚至 500ms 才会去调用 UnlockFunc（执行 Lua 脚本删锁）。
* * * 也就是说，在这接下来的几百毫秒内，Redis 里的坑位依然被 G1 死死占着。
* * 第二幕：G2 盲目冲锋，撞上铁板
* * * [T=4ms] G2 醒了，拿到本地锁，检查 for slot.active。发现大门敞开（是 false），它成功跳出循环。
* * * [T=5ms] G2 按照旧版代码，立刻执行 slot.active = true（把门重新关死），然后兴冲冲地通过网络向 Redis 发送 SetNX。
* * * [T=6ms] G2 的请求到达 Redis。Redis 一看，坑位里还躺着 G1 的专属 Value 呢，无情地返回了 0（抢锁失败）。
* * 第三幕：G2 的“百年孤寂”（被死死卡住）
* * * [T=7ms] G2 收到失败结果，无奈地进入了你的重试循环：
```Go
for i := 0; i < 100; i++ {
    // 抢失败了，进入 select
    select {
    case <-time.After(10 * time.Millisecond): // 死等 10ms
    }
}
```
* * * [T=17ms] G2 等了 10ms，发起第 2 次 SetNX。由于 G1 的业务还没执行完，Redis 再次拒绝，G2 继续睡 10ms。
* * * [T=27ms] G2 发起第 3 次，失败，睡 10ms……
* * * [T=200ms] G2 已经重试了 20 次了，睡了 200ms。直到这个时候，G1 的外层业务终于干完活了，调用了 UnlockFunc，执行 Lua 脚本把 Redis 里的锁删了。
* * * [T=207ms] G2 发起第 21 次重试，终于 SetNX 成功了！

* 灾难的涟漪效应（为什么跑分极差）
* * 你发现了吗？在上面的 T=7ms 到 T=207ms 这整整 200毫秒 的时间里，G2 像个傻子一样在本地进行无效的睡眠和网络 I/O。
* * 更可怕的是，因为 G2 在 T=5ms 的时候，执行了 slot.active = true（把本地城门关了）。这意味着，剩下的 G3 到 G10000 全都被 G2 堵在本地城门外（挂起在 Wait() 里）。
* * 等 G2 历经 200ms 终于拿到锁、干完活，再唤醒 G3 时，G3 又会经历一遍一模一样的“撞墙 -> 等待前任业务干完 -> 拿到锁”的折磨。
* * 10000 个协程，每一个都要在 Redis 门前等前一个人的业务干完。原本可以在 1ms 内并发失败的请求，变成了 10000 个串行执行的几百毫秒，最终的 Benchmark 跑分当然会炸到天际（单次平均耗时 1 秒）。

* isSuccess 拦截器是如何逆天改命的？
* 当我们把 isSuccess 判断加到 for 循环外面后，第二幕的剧情就完全变了：
```go
// 【关键改动】：当被唤醒时，Leader 已经把战报带回来了
if slot.isSuccess {
    slot.mu.Unlock()
    // Leader 成功了，意味着锁被 Leader 占了，作为排他锁，Follower 直接宣告失败，拒绝卷入 Redis！
    return nil, ErrLockFailed
}
```
* * [T=4ms] G2 醒了，跳出 for 循环。
* * [T=5ms] G2 第一眼就看到了 slot.isSuccess == true（G1 留下的战报：兄弟，我已经拿到了排他锁了，你别去了）。
* * [T=6ms] G2 直接在本地执行 return ErrLockFailed。它没有去调 Redis，没有触发 time.After，更没有执行 active = true 去堵门！
* * [T=7ms] 因为门没关，G3、G4...G10000 瞬间全部涌入，看到战报，全部在一瞬间返回失败。
* 在这个改动下，G1 在远端慢吞吞干业务的 200ms 里，剩下 9999 个小弟已经在 1ms 内全部在本地结账走人了，这就叫完美的“降维打击”。

## 第二阶段完整问题清单

以下汇总第二阶段代码（`turbolock_impl.go`）中存在的全部缺陷，按严重度分级：

---

### 🔴 P0 — 功能不可用

#### ① Lua 释放脚本语法错误（首次发现）

```go
const delLuaScript = `
    if redis.call("GET", KEYS[1]) == ARGV[1] then
        return redis.call("DEL, KEYS[1])    // ← "DEL, 缺闭合引号！
    else 
        return 0
    end
`
```

`"DEL, KEYS[1]` 应改为 `"DEL", KEYS[1]`。当前脚本在 Redis 端编译失败，**调用 UnlockFunc 时锁永远无法释放**。压测中 `_ = unlock(ctx)` 吞掉了错误，未被发现。

> 修正：`return redis.call("DEL", KEYS[1])`

---

#### ② isSuccess 永久污染 slot —— 每个 Key 只能成功锁一次（首次发现）

```
时序推演：
  T1: Leader 抢锁成功 → defer 设 isSuccess=true → Broadcast 唤醒 Follower
  T2: Follower 看到 isSuccess=true → 全返回 ErrLockFailed ✓（本轮正确）
  T3: Leader 业务完毕 → UnlockFunc → Redis 锁已删除
  T4: 新请求 → getSlot 拿到同一个 slot
  T5: slot.active=false ✓（已重置）, slot.isSuccess=true ✗（从未重置！）
  T6: 命中 if slot.isSuccess → 直接返回 ErrLockFailed
  → 此后该 Key 永远无法再被任何人获取！
```

**根本原因**：`UnlockFunc` 闭包只操作 Redis，完全没有触及 slot 状态。`isSuccess` 是一个不可逆的单向闩锁。Leader 成功一次后，该 Key 的 slot 永久"中毒"。

> 修正思路：在 UnlockFunc 执行释放后，将 `slot.isSuccess` 重置为 `false`；或在 Lock() 入口新增"锁已过期则清理 isSuccess"的判断逻辑。

---

### 🟠 P1 — 性能瓶颈 / 内存泄漏

#### ③ `time.After` 在重试循环中导致大量堆内存分配（已识别，未修复）

```go
case <-time.After(t.opts.RetryDelay):  // 每次重试都 new 一个 runtime.Timer 对象
```

每次调用 `time.After` 都会在堆上分配新的 `runtime.Timer`。按 Tries=100 配置，单个 Leader 最多泄漏 100 个 Timer 对象。压测数据 934 allocs/op 中相当比例来源于此。

`example/bad_turbolock.go` 已给出正确示范，但 `turbolock_impl.go` 尚未同步修改。

> 修正思路：`timer := time.NewTimer(delay); defer timer.Stop()` + 循环内 `timer.Reset(delay)` 复用。

---

#### ④ globalMu 全局大锁竞争（已识别，搁置中）

```go
func (t *defaultTurboLocker) getSlot(key string) *localSlot {
    t.globalMu.Lock()         // ← 所有不同 Key 抢同一把锁！
    defer t.globalMu.Unlock()
    // ...
}
```

即使两个请求操作完全不同的 Key（如 `lock:order_1` 和 `lock:order_2`），也要排队等 `globalMu`。多 Key 场景下退化为单行道，CPU 上下文切换成本极高。

> 修正思路：将 `map[string]*localSlot + sync.Mutex` 替换为 `sync.Map`（利用其内部的分片无锁读），或自行实现分段锁（如 256 个分片）。

* 选 sync.Map 当前最佳

关键判定：getSlot 的读写比例

```
典型场景推演：
  进程启动 → 业务热点 Key 仅 10~50 个（如秒杀商品 ID）
           → 前 50 次调用：写（创建 slot）
           → 之后百万次调用：纯读（返回已有 slot）
           
  结论：读写比 ≈ 99.99% : 0.01%
```

这个比例恰好是 sync.Map 的设计目标场景——Go 官方文档原文：

```
"The Map type is optimized for two common use cases: (1) when the entry for a given key is only ever written once but read many times … (2) when multiple goroutines read, write, and overwrite entries for disjoint sets of keys."
```

附带收益：为 slot 清理铺路：

sync.Map 自带 Range() 方法用于遍历，天然支持后续清理逻辑。


* 为什么不用分段锁呢？

分段锁适合 写密集 + Key 基数极大 的场景（如全局缓存）。getSlot 的写入只发生在新 Key 首次出现时，之后纯读——分段锁反而多了一层 hash 计算和分片内锁开销，收益为负。

* 为什么不用 RWMutex 呢？

RWMutex 的 RLock()/RUnlock() 仍有原子操作开销。在 10000 协程同时 RLock 时，readerCount 的 CAS 竞争严重。而 sync.Map.Load() 在稳定态是真正的无锁读。

---

#### ⑤ slot map 无限增长 —— 内存泄漏（首次发现）

`slots map[string]*localSlot` 只有 `getSlot` 添加，没有删除逻辑。如果 Key 是动态的（如 `lock:user_{userID}`），每个新用户 ID 都会创建永久的 slot，进程运行越久内存越大。

> 修正思路：给 `localSlot` 添加 `lastUsed` 时间戳，提供 `CleanupSlots(maxAge)` 方法定期清理长时间未使用且非活跃的 slot。


为什么选择  lastUsed 时间戳 ，而不选择引用计数或惰性删除呢？

* 引用计数
* * refCount++ 进 / refCount-- 出，归零可删
* * 性能开销：每次 getSlot + defer 多两次原子操作
* * 引用计数方案的问题：如果某个协程在 getSlot 之后、Lock 之前因为 ctx 超时直接退出了，refCount-- 永远不会被调用，计数泄漏。
* 惰性删除
* * 原理：只在 Lock() 入口发现"残留空槽"时顺手清理
* * 性能开销：零主动开销
* * 惰性删除的问题：slot 引用已经被各处协程持有，Lock 入口清理只能阻止新 Leader，管不到已休眠的 Follower。
* lastUsed 时间戳
* * 原理：记录最后访问时间，超 N 分钟未访问则视为安全
* * 性能开销：每次 getSlot 多一次 time.Now()
* * 时间戳方案的安全边界：

```
假设 maxAge = 5 分钟，Lock() 最长耗时 < 30 秒

安全论证：
  Leader-Follower 一轮完整的生命周期：< 30 秒
  cleanup 只看 > 5 分钟未访问的 slot
  30 秒 << 5 分钟 ⇒ 被删的 slot 上不可能还有存活的 Follower
  
退化场景：如果某个业务持锁超过 5 分钟
  → active=true ⇒ 跳过清理（安全）
```

* * lastUsed 外部调用方使用方式：

```go
go func() {
    ticker := time.NewTicker(5 * time.Minute)
    defer ticker.Stop()
    for range ticker.C {
        n := locker.CleanupSlots(5 * time.Minute)
        if n > 0 {
            log.Printf("turbolock: cleaned %d idle slots", n)
        }
    }
}()
```

* * 与问题 ②（isSuccess 污染）的联动
* * 问题 ② 修复（UnlockFunc 重置 isSuccess）后，slot 的状态机变为：

```
                        ┌──────────────────────────────┐
                        │  业务 UnlockFunc              │
                        │  → isSuccess = false         │
     ┌──────┐  SETNX   ┌──────┐              ┌─────────┴──┐
     │ idle │ ───────► │ busy │ ───────────► │ idle again │
     └──────┘          └──────┘              └────────────┘
                                                 │
                                          lastUsed 超过 maxAge
                                                 │
                                                 ▼
                                           Cleanup 安全删除
```

* * 没有 ② 的修复，isSuccess 永远为 true → active 为 false 但 slot 永远处于"刚忙完"状态 → lastUsed 持续更新 → 永远不被清理。所以 ② 是 ⑤ 的前置依赖。

### 🟡 P2 — 边界缺陷

#### ⑥ ctx.Done() 检查时机偏后（首次发现）

`ctx.Done()` 放在重试循环内部，且在第 1 次 SetNX 之后才检查。如果 ctx 一开始就已过期，仍会发出一次无效的 Redis 调用。

> 修正思路：在进入重试循环前增加一次 `select { case <-ctx.Done(): return nil, ctx.Err(); default: }`。

---

#### ⑦ 重试策略无退避机制（首次发现）

固定间隔 `RetryDelay`，无指数退避（exponential backoff）。在高竞争场景下，多个 Leader 可能在同一节拍点同步重试，形成新一级的"微惊群"。

> 修正思路：引入 `cap(RetryDelay << i, maxDelay)` 指数退避，或引入随机抖动（jitter）。

* 退避要解决的真实问题
* * TurboLock 场景下，不是经典的"多客户端争抢导致惊群"，因为合流层已经保证了每个 Key 只有一个 Leader 在重试。真正的问题更微妙：

```
场景：Leader 抢锁失败，意味着 Redis 里的锁还在前一个持有者手里

  固定 50ms 重试：
    [0ms] SETNX → 失败 → 等 50ms
    [50ms] SETNX → 失败 → 等 50ms  
    [100ms] SETNX → 失败 → 等 50ms
    ...前持有者 200ms 后才释放...
    [200ms] SETNX → 终于成功

  问题：200ms 内发了 4 次注定失败的 SETNX，浪费了 3 次。
```

* * **退避的目标不是减少多个 Leader 之间的碰撞，而是让 Leader 在"明知锁还在别人手里"时，降低无效轮询频率。**

方案对比

* A: 纯指数退避	
* * 原理：delay × 2^i
* * 优势：简单，适应不同持锁时长
* * 劣势：多 Key 的 Leader 可能同频
* B: 纯随机抖动
* * 原理：delay × (1 + rand)
* * 优势：绝对避免同步
* * 劣势：第一次延迟可能过短
* C: 指数退避 + 抖动
* * 原理：delay × 2^i × (1 + rand)
* * 优势：两者兼顾
* * 劣势：代码稍多

* 这里选择 A: 纯指数退避 + 上限
* * 理由：在 TurboLock 当前阶段，方案 C 过度设计了。合流层已经把同 Key 并发降到了 1，不存在"100 个 Leader 同步重试"的场景。纯指数退避就足够了。

|---|--------|------|
| 1 | globalMu 竞争要到第三阶段吗？ | **不应该。** 第三阶段（时间轮）解决的是锁自动续期，与 globalMu 无关。globalMu 应在第二阶段内修复（替换为 `sync.Map`）。 |
| 2 | isSuccess 直接返回 Follower 会影响上层吗？ | **不影响，这是正确设计。** 排他锁语义下，Leader 已持锁，Follower 本就该快速失败。上层业务收到 `ErrLockFailed` 后自行决定重试或放弃。一个 slot 内的所有协程抢的是同一把分布式锁，逻辑上是同一业务操作。 |
| 3 | 为什么 UnlockFunc 不重置 isSuccess？ | **这是一个 P0 级设计缺陷（见上方 ②）。** `isSuccess` 在 Leader 成功后被置为 `true` 且永不重置，导致 slot 永久中毒。需要在 UnlockFunc 或 Lock 入口增加重置逻辑。 |
