# 压测性能数据分析1

go test -bench=BenchmarkTurboLock_HighConcurrency -benchmem

```
goos: linux
goarch: amd64
pkg: turbolock
cpu: Intel(R) Core(TM) i7-7700HQ CPU @ 2.80GHz
BenchmarkTurboLock_HighConcurrency-8         100         142537929 ns/op           57952 B/op       1006 allocs/op
PASS
ok      turbolock       14.307s
```

* 142537929 ns/op（**单次操作耗时 ~142 毫秒**）
* * 大局观解构： 游戏服里一个接口的生命周期通常要求在 5~10 毫秒 以内，你这一个锁就卡了 142 毫秒，在线上就是灾难。
* * 底层原因： 
* * * 8 个核心（-8）带动的海量 Goroutine 毫无拦截地全部通过网络冲向了单台 Redis。
* * * Redis 的单线程处理能力在密集的网络 I/O 拥堵下成了瓶颈。
* * * 更要命的是，大部分 Goroutine 抢锁失败后，进入了 time.After(50*time.Millisecond) 的重试等待。
* * * 网络排队延时 + 固定的重试退避延时，死死拖慢了吞吐量。

* 1006 allocs/op（**单次抢锁成功竟分配了 1006 次内存！**）
* * 大局观解构： 这个数字极其恐怖。高并发系统最怕的就是频繁申请内存触发 Go 的 GC（垃圾回收）停顿。
* * 层原因： 为什么红锁 redsync 没这么夸张，我们单点版却爆了？因为在 b.RunParallel 的极限压测下，绝大多数 Goroutine 都抢锁失败了，然后进入了 for i := 0; i < t.opts.Tries; i++ 循环。
* * * 在循环体内部，每一次重试都在做 t.client.SetNX(...)。go-redis 客户端在发送请求时，会隐式地创建大量的 Cmd 对象、把 string 强转成 slice、以及底层网络库的 buffer 申请。
* * * 每一轮重试的 time.After(50 * time.Millisecond) 都在底层向 Go 的 runtime 注册一个计时器（Timer）对象，这些对象全部逃逸到了堆上！重试 100 次，内存就暴涨 100 倍。 

* 57952 B/op（**单次操作消耗 ~56 KB 内存**）
* * 由于上面那 1006 次频繁的堆内存分配，导致一个轻量级的分布式锁，在频繁重试的过程中活生生挤出了 56 KB 的垃圾。1 万个并发就是 560 MB 的堆内存震荡。
