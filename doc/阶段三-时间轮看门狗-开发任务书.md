## 一、为什么要做阶段三？

### 当前代码的致命漏洞

```
  Leader G1                Redis                  Leader G2
     │                       │                       │
     │── SETNX key TTL=8s ──►│                       │
     │◄────── OK ────────────│                       │
     │                       │                       │
     │ [业务执行中...]        │                       │
     │                       │                       │
     │                 T=8s: TTL 到期                 │
     │                 锁自动删除！                    │
     │                       │◄── SETNX key ──────────│
     │                       │────── OK ─────────────►│
     │                       │                       │
     │  !! 两个协程同时       │                       │
     │    进入临界区 !!       │                       │
     │                       │                       │
     │ T=9s: 业务完成         │                       │
     │── Lua DEL key ───────►│                       │
     │                       │  !! 把 G2 的锁误删了   │
```

**根本原因**：锁只有被动 TTL 过期，没有主动续期。一旦业务执行时间 > `Expiry`，临界区保护彻底失效。

### 阶段三要达成的目标

| 目标 | 说明 |
|------|------|
| 自动续期 | 锁在持有期间，由系统自动向 Redis 发送 `EXPIRE` 刷新 TTL |
| 一锁 NOT 一协程 | 坚决不用 `go func() { time.Sleep() }` 为每个锁起一个续期协程 |
| 全局单协程驱动 | 整个 TurboLocker 实例只用一个 goroutine + 一个 `time.Ticker` 管理所有锁的续期 |
| 生命周期闭环 | Lock 成功 → 挂载到时间轮；Unlock → 从时间轮剥离 |

## 二、核心设计：层级时间轮（Hashed Hierarchical Timing Wheel）

### 2.1 为什么是时间轮？

| 方案 | 单锁开销 | 10000 锁开销 |
|------|---------|-------------|
| 一锁一 goroutine + Sleep | 1 goroutine + 1 Timer | 10000 goroutine + 10000 Timer = **内存爆炸** |
| 全局最小堆（heap） | O(log N) | 10000×log(10000) ≈ 13 万次比较/秒 |
| **层级时间轮** | **O(1)** | **恒定 O(1)，与锁数量无关** |

### 2.2 三层时间轮结构

```
┌──────────────────────────────────────────────────────────────────┐
│                     TimimgWheel（全局单例）                        │
│                                                                   │
│  心跳: time.NewTicker(100ms)，单 goroutine 驱动                   │
│                                                                   │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  Level 0（秒针层）                                            │ │
│  │  ├─ 槽位数: 256                                              │ │
│  │  ├─ 每槽跨度: 100ms                                          │ │
│  │  ├─ 覆盖范围: 0 ~ 25.6 秒                                    │ │
│  │  └─ 光标: cursor0 → 每秒推进 10 格                           │ │
│  └─────────────────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  Level 1（分针层）                                            │ │
│  │  ├─ 槽位数: 64                                               │ │
│  │  ├─ 每槽跨度: 25.6 秒                                        │ │
│  │  ├─ 覆盖范围: 0 ~ 27.3 分钟                                  │ │
│  │  └─ 触发条件: Level 0 走满一圈 → Level 1 推进 1 格           │ │
│  └─────────────────────────────────────────────────────────────┘ │
│  ┌─────────────────────────────────────────────────────────────┐ │
│  │  Level 2（时针层）                                            │ │
│  │  ├─ 槽位数: 64                                               │ │
│  │  ├─ 每槽跨度: 27.3 分钟                                      │ │
│  │  ├─ 覆盖范围: 0 ~ 29.1 小时                                  │ │
│  │  └─ 触发条件: Level 1 走满一圈 → Level 2 推进 1 格           │ │
│  └─────────────────────────────────────────────────────────────┘ │
│                                                                   │
│  理论最大覆盖: 256 × 64 × 64 × 100ms ≈ 29 小时                   │
│  实际需求: 单个锁 TTL 默认 8s，续期间隔 ~2.6s（TTL/3）            │
│  → Level 0 完全覆盖，Level 1/2 为极端场景兜底                      │
└──────────────────────────────────────────────────────────────────┘
```

### 2.3 每个槽位的链表结构

```
  Slot[17]
    │
    ▼
  ┌──────────┐    ┌──────────┐    ┌──────────┐
  │ task A   │───→│ task B   │───→│ task C   │
  │ key="x"  │    │ key="y"  │    │ key="z"  │
  │ cb=renew │    │ cb=renew │    │ cb=renew │
  └──────────┘    └──────────┘    └──────────┘
  
  每个 task：
  - 双向链表节点（prev / next）
  - key: Redis 锁的键名
  - callback: 续期函数（redis.EXPIRE）
  - circleRound: 降级时所在层级（用于级联时定位）
```

### 2.4 核心算法

#### 添加任务（O(1)）

```
addTask(key, interval, callback):
  1. 计算目标 tick 数: ticks = interval / tickDuration
  2. 确定层级: 如果 ticks < 256 → Level 0，否则降级到 Level 1...
  3. 槽位 = (cursor[level] + ticks) % slotsPerLevel[level]
  4. 插入目标槽位的链表头部
  5. 返回 taskID
```

#### 移除任务（O(1)）

```
removeTask(taskID):
  1. 从链表中摘除节点
  2. 释放内存
```

#### 指针推进（每 tick 触发）

```
advance():
  cursor0 = (cursor0 + 1) % 256
  
  // 执行当前槽位的所有到期任务
  for task in slots[0][cursor0]:
      task.callback()  // 向 Redis 发送 EXPIRE
  
  // Level 0 走满一圈 → 降级触发
  if cursor0 == 0:
      advanceLevel(1)
```

## 四、续期策略设计

### 4.1 续期时机

```
TTL = 8s（Expiry）

  ┌──────────┬──────────┬──────────┬──────────┐
  │ 续期     │ 续期     │ 续期     │ 续期     │
  ▼          ▼          ▼          ▼          ▼
  0s        2.6s       5.3s       8.0s       ...
             ↑
        RenewInterval = Expiry / 3
```

每次续期将 TTL **重置为完整的 `Expiry`**（不是续加），确保业务在续期后始终有完整的保护窗口。

### 4.2 续期失败的处理

```
续期失败（网络超时 / Redis 宕机）：
  → 不重试（下次 tick 自然会再次触发）
  → 记录错误日志
  → Redis TTL 到期后锁自动释放（最终安全）
```

### 4.3 安全边界

```
极端场景：业务代码 panic，UnlockFunc 未被调用
  → 续期任务仍在时间轮中
  → 持续向 Redis EXPIRE
  → 锁永远不会释放 ← 这是问题！

缓解方案：
  1. 在 addTask 时记录 maxHoldDuration（如 30s）
  2. 续期回调中检查：如果 time.Since(acquiredAt) > maxHoldDuration
     → 停止续期，让 TTL 自然过期
  3. 可选：配置项 WithMaxHoldDuration
```

## 五、验证标准

### 5.1 单元测试

| 测试用例 | 验证点 |
|---------|-------|
| `TestTimingWheel_AddAndFire` | 添加任务 → tick 推进 → 回调被执行 |
| `TestTimingWheel_RemoveBeforeFire` | 添加任务 → 移除 → tick 推进 → 回调不被执行 |
| `TestTimingWheel_CascadeDegrade` | Level 0 走满一圈 → 任务从 Level 1 降级到 Level 0 |
| `TestTimingWheel_Concurrent` | 并发 add/remove，无 data race，无 panic |
| `TestTimingWheel_Stop` | Stop() 后 ticker 停止，无 goroutine 泄漏 |

### 5.2 集成测试

| 测试用例 | 验证点 |

| `TestAutoRenew_NoGoroutineLeak` | Lock 100 个不同 key → 检查 goroutine 数 ≈ 1（仅时间轮心跳） |
